\section{Champion model (one equation)}

\[
\boxed{
p_g
=
\sigma\!\Bigl(
a\,\operatorname{logit}(p_g^{E})
+
b\,\operatorname{logit}(p_g^{R})
+
c
\Bigr)
}
\]
with \textbf{deployed} stacker coefficients
\(a=0.3592529607156558\),
\(b=0.6407470392843442\),
\(c=0.3253629989522476\),
and
\(\sigma(z)=1/(1+e^{-z})\),
\(\operatorname{logit}(p)=\log(p/(1-p))\).

These deploy coefficients satisfy \(a+b=1\) (temperature \(\tau=1\)): the model
is fitted by penalized maximum likelihood and then \textbf{projected onto a
temperature floor} \(\tau\ge 1\) so it never sharpens duplicate component
information. The raw unconstrained MLE fit was
\(a=0.5696035387238357\),
\(b=1.0159186448349236\),
\(c=0.3132108962839507\)
(\(\tau=1/(a+b)\approx 0.63<1\)); both sets are stored in
\code{artifacts/selected\_spec.json}. See \S\ref{sec:stacker} for the
derivation and rationale.

Component probabilities \(p_g^{E}\) and \(p_g^{R}\) are themselves L2-logistic
calibrations of Elo and (Bradley--Terry + trend) features built with
architecture \texttt{hfa\_75}.

\noindent\textbf{Step 1 --- unconstrained MLE} (no closed form). Let March games
be \(g=1,\ldots,239\). After fitting base models on dates \(<\) 2026-03-01 and
scoring March component probs \(p_g^{E},p_g^{R}\),
\begin{align*}
\hat a,\hat b,\hat c
&=
\arg\max_{a,b,c}
\sum_{g\in\mathrm{Mar}}
\Big[
Y_g\log p_g
+(1-Y_g)\log(1-p_g)
\Big]
-\tfrac12(a^2+b^2),
\\
p_g
&=
\sigma\!\big(
a\operatorname{logit}(p_g^{E})
+b\operatorname{logit}(p_g^{R})
+c
\big).
\end{align*}
Solved by \code{lbfgs} (\code{sklearn}), giving the raw MLE
\((0.5696,1.0159,0.3132)\).

\noindent\textbf{Step 2 --- temperature floor.} The two component logits are
near-duplicates (\(\rho\approx 0.97\)), so the unconstrained fit yields
\(a+b=1.586>1\), i.e.\ temperature \(\tau=1/(a+b)\approx 0.63<1\). A temperature
below one \emph{sharpens} the blend and produced extreme (99\%+) out-of-sample
prices --- a leakage/overfit red flag. The model is therefore deployed with a
floor \(\tau\ge 1\). Preserving the Elo weight \(w=a/(a+b)=0.3593\) and setting
\(a+b=1\):
\[
a \leftarrow \frac{w}{\tau_{\mathrm{floor}}}=0.3593,\qquad
b \leftarrow \frac{1-w}{\tau_{\mathrm{floor}}}=0.6407,
\]
then \(c\) is refit by one-dimensional MLE with \(a,b\) fixed, giving
\(c=0.3254\). The floor is enforced in \code{fit\_logit\_stacker} and pinned by
\code{tests/test\_stacker\_temperature\_floor.py}. Architecture selection still
ranks candidates on the \emph{unconstrained} March log loss; only the deployed
model carries the floor.

\noindent\textbf{How calculated.} For training rows available at that stage,
\[
\hat{\boldsymbol{\beta}}
=
\arg\max_{\boldsymbol{\beta}}
\sum_i
\big[
Y_i\log\sigma(\beta_0+\boldsymbol{\beta}^\top\tilde{\mathbf{x}}_i)
+(1-Y_i)\log(1-\cdots)
\big]
-\tfrac{1}{2C}\|\boldsymbol{\beta}\|_2^2,

with \(\tilde{\mathbf{x}}\) standardized (\code{StandardScaler}),
\(C=C_E\) or \(C_R\), solver \code{lbfgs}.

\subsection{Raw probability and feature}
\[
p_g^{\mathrm{Elo,raw}}
=
\frac{1}{1+10^{-(R_h-R_a+H)/400}},
\qquad
x_g^{E}
=
\frac{R_h-R_a+H}{400}.
\]
Constants: \(R_i(0)=1500\) (Fixed), \(400\) (Fixed), \(H=75\) (Selected).

\subsection{Update with MOV}
\[
R_h' = R_h + K M_g (Y_g-p_g^{\mathrm{Elo,raw}}),\qquad
R_a' = R_a - K M_g (Y_g-p_g^{\mathrm{Elo,raw}}),
\]
\[
M_g
=
\ln(|m_g|+1)
\cdot
\frac{2.2}{\max\bigl(0.25,\,0.001(R_{\mathrm{win}}-R_{\mathrm{lose}})+2.2\bigr)},
\]
where \(R_{\mathrm{win}},R_{\mathrm{lose}}\) are the pregame ratings of the
\emph{winning} and \emph{losing} team (\S\ref{sec:movsign}). Constants:
\(K=10\) (Selected); \(2.2\), \(0.001\), \(0.25\) (Fixed).

\subsection{MOV rating-difference sign (winner \(-\) loser)}
\label{sec:movsign}
The rating difference inside the MOV denominator is the \emph{winner minus
loser} rating, not a fixed \emph{home minus away} rating. This matches the
published FTE form \((\mathrm{ELO}_W-\mathrm{ELO}_L)\) and has two correct
consequences: (i) an \textbf{upset} (a lower-rated team winning) gives a
smaller denominator and hence a \emph{larger} rating update than an expected
result, and (ii) with zero home advantage the update is exactly team-swap
symmetric. Using a fixed home\(-\)away difference instead would make the
autocorrelation correction depend on venue rather than on who actually won,
under-updating home upsets and over-updating home favorites. This behaviour is
pinned by \code{tests/test\_elo\_mov\_winner\_diff.py}.

\subsection{The constant \(2.2\): source, case for use, and how to optimize it}
\label{sec:mov}

\paragraph{Disclosure (external source).}
The value \(2.2\) was \textbf{not estimated from this project's NBA dataset}.
It was adopted from a published, widely cited sports-rating specification:
Nate Silver / FiveThirtyEight,
\textit{Introducing NFL Elo Ratings}
(archived:
\url{https://web.archive.org/web/20171004105151/https://fivethirtyeight.com/features/introducing-nfl-elo-ratings/}),
which defines
\[
M
=
\ln(|PD|+1)
\cdot
\frac{2.2}{(\mathrm{ELO}_W-\mathrm{ELO}_L)\cdot 0.001 + 2.2}.
\]
This submission copies that functional form into
\code{nba\_wp/features.py::\_elo\_multiplier}
when \code{elo\_mov="log"}. Transparency on this point is intentional:
industry models routinely reuse validated structural constants when the
\emph{form} is what matters and the local sample is too short to re-estimate
every shape parameter cleanly.

\paragraph{Professional case for using \(2.2\) here (best case, optimal or not).}
Even though \(2.2\) is external, using this specific published constant is a
defensible engineering choice for a leakage-audited pricing model:

\begin{enumerate}
\item \textbf{Correct economic shape, not an arbitrary fudge.}
The log term gives diminishing returns to blowouts (a 30-point win should not
update ratings like three independent 10-point wins). The fraction involving
\(2.2\) implements the standard underdog/favorite correction so favorites that
``run up the score'' do not inflate permanently. That is precisely the
behavior a book wants in a strength tracker feeding probabilities.
\item \textbf{Stability and reproducibility.}
A public constant with a fixed formula is auditable: two reviewers regenerating
features get the same Elo path. Replacing it with an in-sample tuned value on
one partial season would add a free parameter, widen the selection surface, and
invite overfit to March noise.
\item \textbf{Separation of concerns in the validation design.}
This project deliberately concentrates selection effort on quantities that move
prices most (home advantage \(H\), step size \(K\), BT/trend regularization)
under a March log-loss rule, while holding the MOV \emph{shape} at a known
industry default. That is a standard bias--variance trade: fix low-sensitivity
shape parameters; select high-sensitivity ones.
\item \textbf{Empirical support that the choice is not fragile.}
On this repository's champion architecture, disabling MOV entirely worsens
March log loss (\(\approx 0.4893\) vs \(\approx 0.4880\) with the log-MOV form).
Varying the constant in a neighborhood of \(2.2\) changes March LL only in the
fourth decimal place (flat likelihood). So the \emph{presence} of this MOV
correction is beneficial; the exact published \(2.2\) is a stable operating
point, not a knife-edge claim of global optimality.
\item \textbf{Interpretability for stakeholders.}
\(2.2\) has a clear reading in the literature as the baseline expected
\(\ln(|PD|+1)\) scale in the autocorrelation correction (see Morse's
derivation of \(d=2200=2.2/0.001\)). That is preferable to an opaque
tuned scalar with no external meaning.
\end{enumerate}

\paragraph{How to calculate the value that would have been most optimal.}
If the goal is a \emph{data-optimal} MOV constant \(b^\star\) for this pipeline
(rather than the published \(2.2\)), compute it by the same selection principle
already used for architectures---minimize March sequential log loss---while
holding the champion architecture fixed:

\begin{enumerate}
\item Fix architecture \texttt{hfa\_75} except replace the literal \(2.2\) by a
candidate \(b>0\) in
\[
M_g(b)
=
\ln(|m_g|+1)
\cdot
\frac{b}{\max\bigl(0.25,\,0.001(R_h-R_a)+b\bigr)}.
\]
(Optionally keep the literature ratio by setting the slope to \(b/2200\) instead
of \(0.001\); the one-parameter profile below is the minimal change.)
\item For each candidate \(b\) on a grid
(e.g.\ \(b\in\{1.0,1.5,\ldots,4.0\}\)) or by continuous search:
build features \(\rightarrow\) fit base models on games before 2026-03-01
\(\rightarrow\) fit the March stacker \(\rightarrow\) record March log loss
\(\mathrm{LL}(b)\).
\item Set the optimal replacement
\begin{center}
\fbox{\parbox{0.88\linewidth}{\centering\vspace{0.55em}
\(
\displaystyle
b^{\star}
\;=\;
\operatorname*{arg\,min}_{b}\;
\mathrm{LL}_{\mathrm{March}}(b)
\)
\vspace{0.55em}}}
\end{center}
That is: choose the MOV constant \(b\) that minimizes March sequential log loss.
This \(b^{\star}\) is the March-optimal MOV constant for this model and metric.
On a quick profile of the current code path, \(\mathrm{LL}(b)\) was
minimized toward the high end of the tested grid (near \(b=4\)), with only a
tiny improvement over \(2.2\) (\(\Delta\mathrm{LL}\sim 4\times 10^{-5}\)).
Hence \(2.2\) is not claimed as \(b^{\star}\); it is claimed as a published,
stable default whose March performance is essentially tied with nearby
optima.
\item \textbf{Stronger production estimator (multi-season).}
On a longer NBA history, estimate the literature intercept directly:
regress bin-wise
\(\widehat{\mathbb{E}}[\ln(|PD|+1)\mid\mathrm{win},\,x]\)
on Elo gap \(x\) and take the intercept \(\hat b_0\) (and slope \(\hat a_0\))
as in the Morse/FTE construction, then set \(b\leftarrow\hat b_0\).
Alternatively, nest \(b\) inside the architecture search and select by
prequential log loss with a locked final holdout.
\end{enumerate}

\paragraph{One-sentence position for reviewers.}
\textit{I used the published FiveThirtyEight log-MOV constant \(2.2\) as a fixed,
auditable shape parameter because it encodes a validated blowout/underdog
correction, keeps the Elo path reproducible, and sits on a flat region of the
March likelihood; a March (or multi-season) profile of \(\mathrm{LL}(b)\) is the
correct way to replace it with a data-optimal \(b^\star\) if that becomes a
production requirement.}
